## Overall assessment

The site had a strong visual foundation: a dark neutral palette, one orange accent, restrained typography, compact cards, clear alignment, and real project/game imagery. Home, Projects, Games, Blog, and CV were easy to scan and avoided decorative clutter.

The main gap was between visual polish and interaction maturity. Contrast, keyboard focus, form semantics, recovery states, and custom-widget behaviour were inconsistent. The site worked well as a browsable showcase, but deeper project evaluation and several interactive flows stopped early or relied on implicit behaviour.

## Primary user jobs

1. A recruiter or internship reviewer wants to judge capability and decide whether to shortlist or make contact.
2. A technical peer wants to inspect projects, evidence, trade-offs, and outcomes.
3. A curious visitor wants a quick sense of personality, interests, and taste.
4. A visitor wants to play a game or try an interactive project directly.
5. A visitor wants to ask a question, collaborate, or send a contact message.
6. An agent or API client wants machine-readable context and chat documentation.

The audit found the site strongest at personality, basic contact discovery, and machine-readable reading. It was less complete for recruiter evaluation, technical depth, and game/player task completion.

## Open findings

### 1. Primary action contrast

**Status:** Open · **Priority:** High

The brand orange was `#e8702a`. Measured text/background ratios were:

- `#f0f0f0` heading text on orange: `2.72:1`.
- `#ffffff` white text on orange: `3.10:1`.
- `#000000` black text on orange: `6.78:1`.

The light-text combinations miss the `4.5:1` target for normal text. Affected roles include the Home primary action, footer submit action, and focused skip link.

Revalidate the final foreground/background pair rather than assuming a historical token recommendation is still the right visual choice.

### 2. Secondary text is too dim

**Status:** Open · **Priority:** High

Measured ratios for `#6a6a6a` were:

- On `#0b0b0b`: `3.64:1`.
- On `#151515`: `3.38:1`.
- On `#252525`: `2.83:1`.

Dates, counts, graph labels, metadata, chips, placeholders, and explanatory copy used this token. Many are functional normal text, not decorative text. Separate readable muted copy from truly subtle decoration, and do not use the subtle role for controls or required context.

### 3. Focus is not a coherent system

**Status:** Open · **Priority:** High

Native semantics are present in many places, but links, card links, icon controls, navigation, game cards, summaries, chat controls, and custom SVG nodes depend heavily on hover styling. The People graph explicitly suppresses the default outline.

Every interactive pattern needs an intentional, high-contrast keyboard state. Removing an outline is acceptable only when a visible replacement is guaranteed.

### 4. Contact form semantics and states

**Status:** Open · **Priority:** High

The global footer form has required fields and autocomplete, but placeholders still act as labels. It also lacks field guidance, inline error treatment, a clear pending state, and a first-party success/retry region.

Because the form appears on most pages, this weakness is repeated across the product. Visible labels and persistent recovery information matter more than reducing its apparent visual weight.

### 5. Chat accessibility and recovery

**Status:** Open · **Priority:** High

Chat has open, loading, disabled, rate-limit, and general error copy, but the state model is incomplete:

- Opening does not intentionally focus the input.
- Closing does not return focus to the launcher.
- New messages are not exposed as a polite live log.
- The input uses a placeholder as its only label.
- Failed sends do not offer a clear retry path while preserving intent.
- Escape-to-close and reduced-motion behaviour are absent.
- The floating launcher can obscure lower-right content, especially on mobile lists and custom interaction pages.

### 6. Games state and navigation

**Status:** Partially addressed · **Priority:** Medium-high

The image-led catalog is strong and the cards are large buttons, but they still read more like passive cards than launch controls. The selected game is not reflected in the URL, Back does not restore focus to the originating card, and iframe loading/unavailable states remain minimal.

The current focus transfer into the player is progress. Revalidate it with keyboard and screen-reader users because focusing an iframe is not equivalent to announcing the selected game and mode change.

### 7. People purpose, load, and access

**Status:** Open · **Priority:** Medium-high

The graph/detail/directory concept is distinctive, and native `details` sections provide a usable fallback. However, the page asks first-time visitors to infer its purpose, carries a large dataset without search/filtering, hides keyboard focus on graph nodes, and gives mobile visitors a long explanation before the directory.

Conventional orientation should accompany the graph: a plain-language purpose, visible graph controls, clear selection instructions, searchable/filterable directory content, and a mobile-first path to the directory.

### 8. Project evaluation depth

**Status:** Partially addressed · **Priority:** Medium

Project grouping and tags make breadth easy to scan. Inline detail sections now help where content exists, but the strongest work still needs consistent evidence: problem, role, architecture or hard part, result, screenshots/demo, source/live destination, and lessons or iteration.

Important actions should use descriptive labels and comfortable touch targets. A private-repository marker should not become a dead end when another useful action is available.

### 9. Navigation and state completeness

**Status:** Open · **Priority:** Medium

Desktop navigation exposes the current page, while mobile navigation does not. The mobile menu icon does not clearly change state. The historical audit also questioned whether “People” communicates “influences/interests”; that is a product-language choice requiring validation, not an automatic rename.

Across interactive features, check default, hover, focus-visible, active/pressed, loading, success, error, empty, disabled, selected, and unavailable/private states. Not every control needs every state, but omissions should be deliberate.

### 10. Reduced motion

**Status:** Open · **Priority:** Medium

Global smooth scrolling, chat auto-scroll, transitions, and animated interaction do not account for `prefers-reduced-motion`. Respecting that preference should cover both CSS behaviour and scripted scrolling.

- Mobile should lead users to the directory quickly rather than foregrounding an unavailable graph.

## Design-system implications

The token set covered background, surfaces, borders, muted/dim text, orange, and headings, but interactive UI had outgrown it. Revalidate semantic roles for primary foreground/background, hover/active surfaces, focus, success, warning, danger, and informational feedback.

Shared Button, IconButton, CardLink, Tag, TextInput, Textarea, and external-link patterns would reduce state and contrast drift. Component extraction is useful only when it centralises behaviour and acceptance criteria; it is not a goal by itself.

Typography scale and alignment were not the problem. Low-contrast small text and inconsistent state styling were.

## Priority sequence

1. Re-measure and fix primary-action and functional secondary-text contrast.
2. Establish one visible keyboard-focus treatment and remove unreplaced outline suppression.
3. Give the contact form persistent labels and complete pending/success/error behaviour.
4. Harden chat semantics, focus lifecycle, recovery, motion, and mobile placement.
5. Finish the Games mode transition: explicit action, shareable state, focus return, loading, and unavailable behaviour.
6. Make People understandable without learning the graph; then improve graph keyboard access and directory efficiency.
7. Deepen evidence for the strongest projects and enlarge important project actions.
8. Apply mobile-nav, blog-navigation, 404, and documentation polish after higher-risk interaction work.

## Acceptance checks

### Global and responsive

- Normal-size text and control labels meet `4.5:1`; large text meets the applicable WCAG threshold.
- Functional muted text is readable on every surface where it appears.
- All interactive controls have a clearly visible `focus-visible` state.
- Keyboard order follows the visual/task order, with no focus loss or traps.
- `prefers-reduced-motion: reduce` disables smooth scripted and CSS scrolling/animation where appropriate.
- Primary routes have no unintended horizontal overflow at 390 px and remain usable at 200% zoom.
- Mobile and desktop navigation expose the current page semantically.

### Contact form

- Every field has a persistent programmatic and visible label.
- Validation identifies the field, explains recovery, and preserves entered values.
- Pending, success, error, and retry states are visible and announced appropriately.
- Submission is possible with keyboard and without relying on placeholder text.

### Chat

- Opening focuses the intended control; closing restores focus to the launcher.
- Messages use appropriate log/live semantics without repeatedly announcing history.
- Loading is exposed as busy, failures preserve user intent, and retry guidance is actionable.
- Escape closes the dialog/panel when appropriate.
- The launcher and panel do not obscure required content or controls at mobile widths.

### Games

- Each game has an explicit, descriptive launch action.
- Selected game state is shareable or deep-linkable if the experience is intended to support sharing.
- Entering and leaving player mode moves and restores focus predictably.
- Loading, unavailable, and embed-failure states do not expose raw fallback markup or dead links.
- Player controls remain readable and comfortably targetable on narrow screens.

### People and projects

- Every graph action has a visible keyboard state and a usable non-graph path.
- Page purpose and graph instructions are understandable without experimentation.
- Mobile users can reach the directory without passing through an oversized unavailable-feature explanation.
- Important project actions are descriptive, keyboard-visible, and at least 40 px targets, with 44 px preferred.
- The strongest projects expose enough evidence to answer problem, role, hard part, and result.

### Validation with people

After the objective accessibility fixes, run three to five task-based sessions covering: choosing a backend-relevant project, launching and leaving a game, finding/downloading the CV, sending a message, and finding a software creator on People. Record task completion, errors, hesitation, and recovery rather than treating visual preference as the result.
